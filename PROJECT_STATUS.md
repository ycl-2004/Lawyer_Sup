## 一、当前项目一句话总结

LawDesk Junior 是一个面向初级律师的**劳动争议案件材料整理与仲裁文书草稿工作台**：除两个离线演示案件外，现已支持「**实时案件**」——真实上传 PDF/docx/txt → 真实解析与规则/LLM 抽取 → 确定性计算 → RAG 检索 → 模板草稿，全部结果**持久化在本地 SQLite**（14 天无活动自动清理、活动自动续期），并提供面向非技术用户的双击一键启动。它已是**本地可用版本**，距正式交付还缺评估数字、法条官方核对与交付材料。

## 二、当前项目整体状态

**判定：已有可用版本（本地单机），接近完整交付的前一阶段。**

理由（对照 v3 的三个硬条件）：
- ✅ AI 抽取链路真实化：上传文件被真实解析（PyMuPDF/python-docx/txt，多编码兜底），抽取采用「规则正则打底（确定性、离线可用）+ DeepSeek LLM 增强（配 key 即启用，引用逐字校验）」；扫描件/图片有「手动粘贴文本」兜底。
- ✅ 持久化：实时案件全量落 SQLite（含确认审计记录），刷新/重启不丢；14 天 TTL 级联清理已测试。
- ✅ 前后端数据同步：工作流运行后前端从 `GET /workspace` 拉取全量数据包渲染。
- 尚未到「正式交付」：抽取评估集与准确率数字缺失、法条未经官方核对、无 CI、交付/验收材料未做、人工编辑（时间线/请求项增删）在实时案件中尚未回写后端。

## 三、当前已经做得好的地方

**已实现且能跑通的功能**
- Dashboard（案件列表/统计/动态/风险提醒）+ 三栏案件工作台（7 个工作区 + AI Inspector）。
- 两个完整模拟案件：张某违法解除案（默认载入）、刘某加班费案（经工作流动画载入）。
- 材料管理：模拟上传、人工改分类、点击行预览全文+分类依据+关联事实、删除（带确认）。
- 可编辑时间线（增/删/标记争议）、请求项（增/删/采纳/排除、折叠详情）、缺失材料清单（增/删）。
- **确定性金额计算服务（项目技术核心，TS+Python 双实现）**：拖欠工资、经济补偿（47条年限推导+三倍封顶+12年上限）、违法解除赔偿（×2）、二倍工资差额（11个月封顶）、加班费（21.75基数）、仲裁时效检查。缺输入即拒绝计算，公式/推导/依据全程可见。
- **法条 RAG（中国劳动法）**：20 条条文级语料（带生效日期/官方出处/关键词），BM25+TF-IDF+关键词混合检索，前后端同算法（后端不可达自动回退），golden 查询 10/10；草稿引用反幻觉核对（能识别幻觉法律名与幻觉条号）。
- FastAPI 后端骨架：mock/DeepSeek 双模式 LLM 客户端、事实抽取 Agent（引用必须逐字出自原文否则强制 needs_review）、6 节点工作流管线、8 个 API。
- 导出：docx（首页强制红色草稿水印）、Markdown 材料包（含免责声明/确认审计记录/占位符计数）。
- 风控与合规设计贯穿：草稿水印、来源锚点、缺失占位符不补全、三档置信度（避免伪精确）、数据层脱敏（身份证/手机号/账号正则）、律师确认门槛（高风险项全部处理+签名才可导出）、确认审计记录（人+时间）。
- 可访问性基线：focus-visible、cursor-pointer、prefers-reduced-motion。

**可证明"不是空想"的证据**
- 测试：前端 vitest 23 项 + 后端 pytest 14 项 + RAG golden 10/10（前后端排序一致）+ 脱敏泄漏检测，全部通过（沙盒实测记录）。
- 可复现：`npm install && npm run dev` 前端即跑；后端 `pip install -r requirements.txt && uvicorn` 即跑（mock 模式零外部依赖）。
- 代码结构：约 50 个前端文件 + 18 个后端文件，分层清晰（types/data/lib/state/components/pages；api/agents/rag/services/workflows）。

**可直接用于展示/作品集的素材**
- 「刘某案加班小时为空 → 计算卡片拒绝计算 → 人工输入后出金额」——展示"系统不推测"红线的最佳演示。
- 「引用核对抓出《不存在法》和第九百条」——反幻觉能力的直观证据。
- 工作流运行弹窗（后端模式绿色徽章 + 真实节点摘要）。
- 计算卡片展开的推导过程（法条依据逐步可见）。

## 五、风险分析

**1. 技术风险（中）**
代码本身结构清晰、依赖少（前端 4 个运行时依赖）。主要风险：(a) `docx` 库导出在不同 Word 版本的兼容性未实测；(b) 前端 `tsc --noEmit` 与 `vite build` 只在用户机器跑过 dev 模式，production build 未验证；(c) 后端 FastAPI 层在沙盒中无法安装依赖，仅服务层经过测试——uvicorn 启动后的接口行为靠人工验证过一次。

**2. 产品风险（中高）**
定位（律师内部辅助、不替代判断）是清晰且站得住的。最大风险是「**演示价值 ≫ 实用价值**」：真实律师拿散乱 PDF 来用时，系统目前什么都抽不出来。第二风险：劳动争议计算各地口径差异大（社平工资、应发/实发、仲裁委格式），当前用"人工输入+警告"兜底是诚实的做法，但意味着真实效率提升有限。结论：作为课程项目/作品集是强的，作为产品还需验证「AI 抽取准确率是否高到真省时间」。

**3. 用户体验风险（低中）**
本轮已修复信息过载、无预览、无增删等问题。残留：流程对首次使用者仍需引导（建议加 onboarding 提示或演示脚本）；窄屏体验靠浮动按钮兜底，未做真正移动端适配；中文以外无国际化（按定位可接受）。

**4. 数据/隐私/安全风险（演示态低、投产态高）**
当前全部数据为虚构，无真实风险。但架构上：脱敏只在前端（正则兜底，可绕过）；无传输/存储加密；无访问控制；API 无鉴权（CORS 仅限 localhost）；如接真实客户材料，PIPL 合规（最小必要、删除权、审计）全部未实现。**在接入任何真实材料前，这些是硬性前置条件。**

**5. 交付风险（中）**
有 README、运行步骤、测试命令——比多数课程项目好。缺：验收 checklist、截图/录屏存档、测试输出的留档文件、CHANGELOG、明确的"验收人操作脚本"（按什么顺序点什么、应该看到什么）。验收者如果不按 README 走（例如没装 Node 18+ / Python 3.10+）会卡住。

**6. 内容表达风险（高，须重点注意）**
绝对不能写的：「AI 自动完成案件分析」「已接入 DeepSeek 实际运行」（接口就绪≠已验证真实抽取质量）、「法条库权威/完整」、「可投入律所使用」、任何暗示替代律师或预测结果的表述。
必须保守表达的：「演示数据」「条文样例需核对官方文本」「抽取链路为模拟，架构已就绪」「无真实用户验证」。
可以放心说的：确定性计算（有测试）、反幻觉引用核对（有证据）、混合检索 golden 10/10、风控设计体系、前后端双实现一致性。

**7. 时间风险**
最耗时的三件事（按序）：① 真实 LLM 抽取的质量调优（prompt + 校验 + 评估集，预计占剩余工时 40%）；② 持久化 + 前后端数据同步（状态模型要从"预置数据集"重构为"服务端状态"）；③ 法条文本官方核对（机械但不能省）。容易卡住的：DeepSeek API 在长文档上的 JSON 稳定性、PDF 解析的中文表格。应后置的：登录权限、多地区参数、向量库升级、移动端。
